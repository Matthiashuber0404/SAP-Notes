1. PBO und PAI

Zentral bei der Dialogprogrammierung ist die Programmierung eines Dynpros. Zum Dynpro gehört die Definition der Felder auf dem Dynpro und die zugehörige Ablauflogik.

Es gibt in der Ablauflogik zu einem Dynpro vier Ereignisse:

  • PBO (Process Before Output)

  • PAI (Process After Input)

  • POV (Process On Value-request)

  • POH (Process On Help-request)

In der Ablauflogik gibt es zunächst lediglich einen sehr begrenzten Befehlsumfang. Es ist z. B. nicht möglich, direkt unter dem Befehl "process before output" einen PERFORM einer Form-Routine zu schreiben.

Um in der Ablauflogik das Coding zu modularisieren, wird mit dem Befehl MODULE ein Modul angelegt. Wobei dann innerhalb des Moduls der komplette ABAP-Befehlsumfang zur Verfügung steht.

Der Modulname wird durch "module" begonnen. Es folgt der Modulname und es schließt mit "output" beim PBO, bzw. "input" beim PAI. Das Coding des Modules schließt mit "module."

Das Ereignis "Process Before Output" (PBO) wird ausgelöst, bevor das Dynpro aufgerufen wird, bzw. auf dem Bildschirm erscheint. Hier finden normalerweise mehrere Dinge statt:

  • Zuweisung eines Status zum Dynpro

  • Zuweisung eines Titels zum Dynpro

  • Datenbefüllung der Felder vom Dynpro

Das Ereignis "Process After Input" (PAI) wird ausgelöst bei einer Aktion vom Benutzer auf dem Bildschirm:

  • Drücken einer Schaltfläche auf dem Dynpro

  • Drücken eines Symbolleisteneintrags

  • Auslösen einer Tastenkombination, die dem Dynpro bekannt ist (z. B. ENTER)

  • Auswählen eines Menüeintrages, der im Dynpro angelegt wurde.

Damit in der Anwendungslogik bekannt ist, welche Aktion den PAI ausgelöst hat, wird der sogenannte Funktionscode dem Code im PAI übergeben.

In der Systemvariablen SY-UCOMM steht der Wert des Funktionscodes. Jede entsprechende Aktion auf dem Dynpro muss in seinen Attributen einen eindeutigen Funktionscode haben.

Die Übergabe der Daten an ein Dynpro erfolgt über namensgleiche Felder. Die Variablen müssen in der Ablauflogik dabei global definiert sein. Sie werden in der Ablauflogik mit dem Befehl TABLES definiert oder man arbeitet mit eigenen Strukturen.

2. Dynpro Control SALV-Grundmodell

Der SALV ist ein Instrument zur zweidimensionalen Darstellung von Daten. Die Zeilenanzahl kann beliebig sein, man benötigt jedoch eine konstante Struktur.

Dazu gibt es die bereits bestehende Hauptklasse CL_SALV_TABLE.

Der SALV kann im ganz normalen Report und im Dynpro verwendet werden und besitzt eine Menge an Methoden. Zunächst werden folgende Methoden vorgestellt: factory (statisch), display, refresh.

2.1. Verwendung im Report

Die Erzeugung eines einfachen SALV geschieht in vier Schritten. Als Beispiel soll die bereits bestehende DB-Tabelle pupil verwendet werden (Z001_63_PUPIL).

(a) Datendefinitionen
Zunächst muss eine interne Tabelle erstellt werden, deren Definition zwei Funktionen aufweist.

  • Sie enthält die Daten, die an das SALV-Werkzeug übergeben werden.

  • Sie wird vom SALV-Werkzeug zum Erstellen für den SALV verwendet.

"1.Definitionen
"a) Struktur und itab
TYPES: BEGIN OF ts_pupil,
         surname TYPE z001_63_pupil-surname,
         name    TYPE z001_63_pupil-name,
         city    TYPE z001_63_pupil-city,
         cid     TYPE z001_63_pupil-cid,
       END OF ts_pupil.
DATA: gt_pupils TYPE TABLE OF ts_pupil.
"b) Referenzvariable
DATA: go_pupil TYPE REF TO cl_salv_table.

Die Referenzvariable go_pupil wird für den Zugriff auf die Unterklassenobjekte des SALV-Werkzeugs verwendet.

(b) Interne Tabelle füllen

"2.Interne Tabelle füllen
  SELECT surname name city cid
        FROM z001_63_pupil
        INTO TABLE gt_pupils.

(c) Objektinstanzierung

Zur Objekterstellung muss die statische Methode factory( ) der Klasse CL_SALV_TABLE aufgerufen werden.

"3.SALV erzeugen
TRY.
    CALL METHOD cl_salv_table=>factory
      EXPORTING
        list_display = if_salv_c_bool_sap=>true   "ALV im Listenmodus anzeigen
      IMPORTING
        r_salv_table = go_pupil
      CHANGING
        t_table      = gt_pupils.
  CATCH cx_salv_msg INTO DATA(oerror).
    MESSAGE oerror->get_text( ) TYPE 'E'.
ENDTRY.

Obige Darstellung setzt den List-Modus. Der EXPORTING-Parameter list_display muss auf ABAP_TRUE gesetzt werden, wenn die Ausgabe als klassische ABAP-Liste erwünscht ist.

(d) Anzeige

Im Anschluss an den Aufruf der FACTORY-Methode muss die Methode display( ) aufgerufen werden, um die Ausgabe anzuzeigen.

 "4.SALV ausgeben
  go_pupil->display( ).

2.2. Verwendung im Dynpro

Soll ein SALV im Dynpro verwendet werden, dann ist folgendermaßen vorzugehen:
(a) Dynpro 100 anlegen
(aa) Anlegen eines Containers im Dynpro:
Wählen Sie dazu die CustomControl-Taste und ziehen Sie diese über den gewünschten Bereich auf. Anschließend geben Sie diesem einen Namen (CC_PUPILS)

(ab) Weitere Felder können normal im verbliebenen Platz des Dynpros angelegt werden. Natürlich wird auch ein OK_CODE verwendet.

(ac) Ablauflogik erstellen:

PROCESS BEFORE OUTPUT.
  MODULE status_0100.
  MODULE read_data_0100.
  MODULE create_salv_0100.
*
PROCESS AFTER INPUT.
  MODULE user_command_0100.

(b) Weitere (zusätzliche) Definitionen im TOP-Include: Damit der Container im Dynpro den SALV aufnehmen kann, muss dafür eine Referenzvariable definiert werden.

TYPES: BEGIN OF ts_pupil,
         surname TYPE z001_63_pupil-surname,
         name    TYPE z001_63_pupil-name,
         city    TYPE z001_63_pupil-city,
         cid     TYPE z001_63_pupil-cid,
       END OF ts_pupil.
DATA: gt_pupils TYPE TABLE OF ts_pupil.
DATA: go_pupil TYPE REF TO cl_salv_table.

DATA: ok_code      TYPE sy-ucomm,
      go_container TYPE REF TO cl_gui_custom_container.

(c) Das Füllen der internen Tabelle muss genauso wie bei der Anzeige im Report durchgeführt werden. (module read_data_0100)

SELECT surname name city cid
          FROM z001_63_pupil
          INTO TABLE gt_pupils.

(d) Container und SALV erzeugen, SALV ausgeben (MODULE create_salv_0100)

  " Containerobjekt
  IF go_container IS INITIAL.
    go_container = NEW cl_gui_custom_container(
        container_name              = 'CC_PUPILS'
    ).
  ENDIF.
  " SALV-Objekt
  TRY.
      CALL METHOD cl_salv_table=>factory
        EXPORTING
          r_container    = go_container
          container_name = 'CC_PUPILS'
        IMPORTING
          r_salv_table   = go_pupil
        CHANGING
          t_table        = gt_pupils.
    CATCH cx_salv_msg INTO DATA(oerror).
      MESSAGE oerror->get_text( ) TYPE 'E'.
  ENDTRY.
  go_pupil->display( ).

2.3. Manipulation des SALV

Es gibt eine Vielzahl an Unterklassen zur Manipulation. Zum Zugriff verwendet man eine der verschiedenen GETTER-Methoden der Hauptklasse.

2.3.1. Funktionsleiste

Hinzufügen der standardmäßigen Funktionsleiste (Verwendung von get_functions( )):

Diese ist ein sehr mächtiges Tool und erlaubt auf- und absteigendes Sortieren, Filtern, Summieren, Suchen, Export, usw. Die Funktionsleiste wird oberhalb der SALV-Tabelle hinzugefügt und schaut dann folgendermaßen aus:

Figure 4. Funktionsleiste SALV

Fügen Sie nach Erzeugen des SALV (nach der erfolgreichen Durchführung der factory-Methode und vor der display-Methode folgende Anweisung aus:

go_pupil->get_functions( )->set_all(
    value = IF_SALV_C_BOOL_SAP=>TRUE
).

Erläuterung:
get_functions( ) ist eine Instanzmethode der Klasse CL_SALV_TABLE und erzeugt durch den Aufruf eine Instanz der Klasse CL_SALV_FUNCTIONS. Diese Klasse besitzt die Instanzmethode set_all( ), mit der Sie alle vorhandenen Standardfunktionen aktivieren und eine Statusleiste erzeugen.

2.3.2. Spaltenattribute

Bearbeiten von Spaltenattributen (Verwendung von get_columns( )):

Spalten bearbeiten können Sie mit Hilfe der Unterklassen CL_SALV_COLUMNS_TABLE (Erzeugen der Objektreferenz mit der Methode get_columns( ) zum Manipulieren von Spalten) und CL_SALV_COLUMN (Holen gewünschte Spalte, Manipulieren der Eigenschaften).

Um Spalten zu bearbeiten, müssen Sie in zwei Schritten vorgehen:

1) Zuerst muss die Objektreferenz für die Klasse CL_SALV_COLUMNS_TABLE mithilfe der Methode get_columns( ) abgerufen werden.

2) Um Spalten zu bearbeiten, muss die Objektreferenz der Klasse CL_SALV_COLUMN über den Methodenaufruf erzeugt werden (für diese Spalte!!). Anschließend können beispielsweise über set_output_length( 15 ) und set_alignment( value = 3 ) die Attribute bearbeitet werden.

Fügen Sie folgende Anweisungen wieder nach der factory-Methode ein!

go_pupil->get_columns( )->get_column( 'CITY' )->set_output_length( value = 15 ).
go_pupil->get_columns( )->get_column( 'CITY' )->set_medium_text( value = 'Wohnort' ).
go_pupil->get_columns( )->get_column( 'CID' )->set_output_length( value = 7 ).
go_pupil->get_columns( )->get_column( 'CID' )->set_alignment(  value = 3 ).

2.5. Event beim SALV

In der Unterklasse CL_SALV_EVENTS_TABLE sind bereits die Ereignisse double_click (Doppelklick) und link_click (Einfachklick) definiert und können einfach verwendet werden.

Note
 Parameter für Einfachklick sind identisch mit denen des Doppelklicks.
Figure 6. Klasse cl_salv_events_table
Figure 7. Parameter der Methode double_click

Behandlerklasse definieren

Da bereits Events in obiger Klasse existieren und ausgelöst werden, können diese einfach verwendet werden. Daher ist nur noch eine Behandlerklasse nötig:

CLASS lcl_handle_events DEFINITION.
  PUBLIC SECTION.
    METHODS:
      on_double_click FOR EVENT double_click OF cl_salv_events_table IMPORTING row column,
      on_link_click FOR EVENT link_click OF cl_salv_events_table IMPORTING row column,
      get_line RETURNING VALUE(rn_no) TYPE i.

  PRIVATE SECTION.
    DATA: mn_line TYPE i.
ENDCLASS.

CLASS lcl_handle_events IMPLEMENTATION.
  METHOD on_link_click.
    mn_line = row.
    cl_gui_cfw=>set_new_ok_code( EXPORTING new_code = 'CLICK'.
  ENDMETHOD.

  METHOD on_double_click.
    mn_line = row.
    cl_gui_cfw=>set_new_ok_code( EXPORTING new_code = 'CLICK'.
  ENDMETHOD.

  METHOD get_line.
    rn_no = mn_line.
  ENDMETHOD.
ENDCLASS.

Ereignisse registrieren

Nach Erzeugung des SALV muss ein Objekt des Eventhandlers erzeugt werden. Darauf ist dann die entsprechende Behandlermethode zu registrieren (SET HANDLER …).

DATA: go_pupil TYPE REF TO cl_salv_table.
DATA: go_event_handler TYPE REF TO lcl_handle_events.
CALL METHOD cl_salv_table=>factory
  EXPORTING
    r_container = NEW cl_gui_custom_container( 'CC_CONT' )
  IMPORTING
    r_salv_table = go_pupil
  CHANGING
    t_table = gt_pupils.

DATA(go_events) = go_pupil->get_event( ).
go_event_handler = NEW #( ).

"Doppelklick registrieren
SET HANDLER go_event_handler->on_double_click FOR go_events.

"Einfachklick registrieren
DATA: go_columnx TYPE REF TO cl_salv_column_table.
go_columnx ?= go_pupil->get_columns( )->get_column( 'NAME' ).
go_columnx->set_cell_type( if_salv_c_cell_type=>hotspot ).
SET HANDLER go_event_handler->on_link_click FOR go_events.

go_pupil->display( ).

Klick-Event verarbeiten

Wird ein Klick auf den SALV durchgeführt, dann wird die entsprechende Methode aufgerufen und die geklickte Zeile in der Instanz gespeichert. Diese Zeilennummer kann dann über die Instanzmethode get_line abgerufen und damit aus die interne Tabelle der entsprechende Datensatz gelesen werden.

WHEN 'CLICK'.
  DATA(gn_no) = go_event_handler->get_line( ).
  DATA(gs_pupil) = gt_pupils[ gn_no ].
  DATA(gv_klick) = |{ gs_pupil-vorname } { gs_pupil-name }|.
