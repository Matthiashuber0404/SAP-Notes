1. Die Historie von ABAP

ABAP ist eine Programmiersprache, die im Jahr 1983 erschienen ist. Es handelt sich dabei um eine sogenannte proprietäre Sprache, was bedeutet, dass sie nicht - wie zum Beispiel Java - einfach genutzt werden kann. Einfach genutzt bedeutet in diesem Zusammenhang, dass es nicht möglich ist eine IDE herunterzuladen und dort mit der Programmierung zu starten. Der Grund dafür liegt darin, dass ABAP die hauseigene Programmiersprache der SAP ist und infolgedessen auch nur in Verbindung mit einem SAP-System genutzt werden kann. Aber was genau ist eigentlich SAP?

SAP ist die Abkürzung für "Systeme, Anwendungen, Produkte in der Datenverarbeitung" und ist ein deutscher Softwarekonzern mit Hauptsitz in Walldorf bei Heidelberg. Das Unternehmen wurde im Jahr 1972 von Claus Wellenreuther, Hans-Werner Hector, Klaus Tschira, Dietmar Hopp und Hasso Plattner fünf ehemalige IBM-Mitarbeiter gegründet. Der Softwaregigant spezialisiert sich dabei darauf, dass allgemeine Geschäftsprozesse in Unternehmen softwaretechnisch optimal abgebildet werden und auch integriert werden können. Seit mittlerweile 45 Jahren und mit über 84.000 Mitarbeiter weltweit, stellt das deutsche Unternehmen dabei einen nicht mehr wegzudenkenden Softwarelieferanten dar. (SAP, 2017) (Färber & Kirchner, 2008)

Sämtliche Module, welche man innerhalb eines SAP-Systems dabei findet, wurden mit ABAP entwickelt. Die Abkürzung ABAP stand ursprünglich für "Allgemeiner Berichtaufbereitungsprozess". Haupteinsatzzweck war zu Beginn das Schreiben von Auswertungen, den sogenannten "Reports". Da sich allerdings die Sprache ab 1983 ständig weiterentwickelt hat, kamen mit neueren Versionen auch immer neue Möglichkeiten hinzu, weshalb der Name auch umgeändert wurde auf "Advanced Business Application Programming". Typische Erweiterungen waren hierbei z. B. die Objektorientierung in Form der ABAP Objects, aber auch allgemeine Datenbankmanipulationen. (ABAP, 2017)

2. Wesentliche Eigenschaften der Programmiersprache

  • 4GL Sprache
    ABAP ist eine Programmiersprache der vierten Generation. Diese Sprache ist daher auf eine sehr schnelle und zügige Verarbeitung von Massendaten ausgelegt. Für diese Verarbeitung sollten dabei weiterhin auch möglichst wenig Codezeilen aufgewendet werden. (4GL, 2017)

  • Open SQL
    Wie auch schon bei ABAP selbst, handelt es sich bei dieser Form der "Structured Query Language" ebenfalls um eine proprietäre Sprache, die von der SAP SE entwickelt wurde. Die SQL Befehle selbst sind dabei direkt in die ABAP Sprache integriert. Hintergrund ist der, dass eine einheitliche SQL Sprache für alle SAP-Datenbanken definiert werden soll. (OpenSQL, 2017)

  • SAP-Pufferung
    Pufferung betrifft hier vor allem Datenbanktabellen. Um nicht ständig erneut komplette Tabellen und deren Inhalt auszulesen, können Zeilen einer Tabelle oder eines Views in den Puffer geladen werden. Dadurch greifen Open SQL Anweisungen nicht direkt auf die Datensätze, sondern vielmehr auf diejenigen, die gepuffert im Shared Memory liegen, zu. Der Shared Memory ist dabei ein spezieller Speicherbereich, in den Programme, Programmdaten, Datensätze etc. von der ABAP Laufzeitumgebung abgelegt werden können. (SAP-Pufferung von Datenbanktabellen, 2017) (Shared Memory, 2017)

  • Interne Tabellen
    Interne Tabellen sind ähnlich wie herkömmliche Datenbanktabellen, nur mit dem Unterschied, dass sie zur Laufzeit generiert werden und auch nur solange existieren, bis das jeweilige Programm abgeschlossen ist. In ihnen können letztendlich Daten dynamisch gespeichert und bearbeitet werden. (Interne Tabellen, 2017)

  • Online Transaction Processing
    OLTP ist das Gegenstück zur sogenannten Batch-Verarbeitung. Bei OLTP stehen dabei Transaktionen im Vordergrund, die gleichzeitig ausgeführt werden können. Das bedeutet, dass es keine Probleme gibt, wenn mehrere Benutzer gleichzeitig auf einen Datenstamm zugreifen wollen. (Online Transaction Processing, 2017)

  • Remote Function Calls
    Häufig existieren mehrere unterschiedliche Systeme. Auf diesen Systemen sind dabei die unterschiedlichsten Programme und Anwendungen. Um nicht der Gefahr von Redundanz zu laufen, bietet SAP die Möglichkeit von sogenannten RFCs an, womit Funktionen auf entfernten Systemen aufgerufen werden können. Das entfernte System muss dabei kein SAP-System sein. (Remote Function Call, 2017)

  • XML-Schnittstelle
    Die "extensible markup language" ist ein gängiges Format, womit Daten zwischen unterschiedlichsten Systemen ausgetauscht werden können. Aus diesem Grund existieren Möglichkeiten in ABAP um XML zu lesen, aber auch zu erstellen.

  • Prozedurale / objektorientierte Abarbeitung
    In seiner Ursprungsform bietet ABAP die Möglichkeit mit Hilfe von Unterprogrammen und Funktionsbausteinen eine komplett prozedurale Programmierweise, ganz ohne Klassen etc. zu erreichen. Wie bereits angesprochen, wird die Sprache allerdings ständig erweitert. So kam mit Release 6.10 ein objektorientiertes Programmiermodell namens "ABAP Objects".

  • Abwärtskompatibilität
    Mit neuen Versionen / Releasen finden natürlich auch ständig neue Sprachelemente den Einzug in die Sprache. Es kann also durchaus vorkommen, dass ein alter Befehl durch eine neue Anweisung ersetzt wird. Diese haben dabei meistens Vorteile bezüglich Performance. Alte Befehle verlieren hierbei natürlich nicht ihre Gültigkeit. Dadurch wird sichergestellt, dass durch Releasewechsel auch ältere Kundenentwicklungen ihre Gültigkeit behalten und nicht überarbeitet werden müssen.

3. Systemarchitektur und erste Schritte im System

Im Jahr 1973 erblickte das erste Enterprise-Resource-Planning System "R/1" das Licht der Welt. Ursprünglich handelte es sich bei diesem ersten System lediglich um ein Modul für Finanzbuchhaltung (System RF), allerdings wurde dies schnell um weitere Module erweitert bzw. ergänzt. Technische Grundlage für das System bildeten dabei Großrechner von IBM und das Betriebssystem DOS. Auf diesen Rechnern fand man dabei Datenbank-, Applikations- und Präsentationsschicht. Es handelte sich also in allen Konstellationen immer um ein einziges System, welches Herr aller drei Schichten war. Deshalb auch die "1" innerhalb des Produktnamens. Das "R" steht dabei grundsätzlich für "Realtime". Da sich Hardware und Rechnertechnologie natürlich ständig weiterentwickelten, fanden in den 80igern langsam aber sicher preisgünstigere und leistungsstärkere Serveranlagen Einzug in den Markt. Für die SAP war es damals an der Zeit ihre Architektur zu überarbeiten. Gründe hierfür waren auf der einen Seite, dass es möglich war die Software selber auf komplett eigenen Serverlandschaften zu betreiben und dadurch mehreren Benutzern gleichzeitig Zugriff zum System zu gewähren. Auf der anderen Seite spielte natürlich die Wartbarkeit eine große Rolle. Nur noch eine einzige Instanz auf der letztendlich das Programm installiert ist, Updates eingespielt werden können und eine zentrale Datenverwaltung stattfindet, erleichtert den Betrieb erheblich. Da nun letztendlich zwei Schichten im Einsatz waren, änderte sich der Name von "R/1" auf "R/2". Folgende Abbildungen veranschaulichen dabei die genaue Zusammenstellung: (Färber & Kirchner, 2008)

Abbildung 1. R/1 Architektur (Färber & Kirchner, 2008)

Wie auch in allen Bereichen der Informationstechnologie, verändert sich Technik und Architektur im Laufe der Zeit. Das alte "R/2" wurde dabei Anfang der 1990er Jahre abgelöst von "R/3". Wie der Name schon vermuten lässt, wuchs die Architektur um eine dritte Schicht, den sogenannten "Clients". Clients stellen herkömmliche Arbeitsplatzrechner dar, die mit grafischen Userinterfaces arbeiten. "R/3" stand allerdings nicht nur für die Aufteilung in drei wesentliche Schichten, sondern auch für die entsprechenden betriebswirtschaftlichen Anwendungen (Business Modules). Zu solchen Anwendungen gehören dabei zum Beispiel Module wie Sales & Distribution (SD), Finance (FI), Controlling (CO), Human Resources (HR) oder Production Planning (PP). (Färber & Kirchner, 2008)

Abbildung 3. R/3 Architektur (Färber & Kirchner, 2008)

Den vorletzten Architekturwechsel betrieb die SAP dabei in den frühen 2000er Jahren, wobei man hierbei Architekturwechsel nur indirekt sagen kann. Die bestehende Architektur wurde vielmehr um einige wesentliche Internettechnologien und Java erweitert. Der Name wechselte dabei von "R/3" auf "mySAP.com". Dieser Name wurde jedoch relativ schnell erneut abgewandelt auf "mySAP" bis schließlich Mitte der 2000er Jahre der vorerst letzte Namenswechsel stattfand auf "SAP ERP". Ein sehr wichtiger und relevanter Architekturwechsel, der auch aktuell den letzten darstellt, ist die Einführung von "SAP S/4 HANA" im Jahr 2015. Hierbei handelt es sich um die komplette Integrierung der In-Memory-Datenbank "SAP HANA" in ein SAP-System. Mit diesem Releasewechsel fand auch eine neue Oberflächentechnologie auf Basis von HTML5, CSS3 und JavaScript namens "SAPUI5" Einzug. Ebenfalls wurde die Datenanbindung an dieser Stelle für die neuen Webanwendungen mit Hilfe des "SAP Netweaver Gateways" überarbeitet. (Färber & Kirchner, 2008)

3.1. Datenbank-, Applikations- und Präsentationsschicht

Jeder "SAP NetWeaver Application Server" folgt dem Client-Server-Prinzip. Dadurch ist es möglich, dass die Präsentations-, die Applikations- und Datenbankebene auf unterschiedlichen Systemen verteilt werden können. Hierdurch ergibt sich natürlich auch wieder ein erheblicher Vorteil für die Skalierbarkeit. Zusätzliche Hardwarekomponenten für Speicherplatz oder auch Rechnerleistung können so zum Beispiel ohne große Probleme erweitert werden. (SAP, 2014)

  • Datenbankschicht
    Wie der Name schon andeuten lässt, sind in dieser Schicht die eigentlichen Daten angesiedelt. Abhängig davon welchen Release eines Systems man nutzt, handelt es sich dabei um eine relationale Datenbank oder um die neuere In-Memory-Datenbank. Gelagert werden hierbei herkömmliche Anwendungsdaten, Metadaten, aber auch Programme für die eigentliche Verwaltung dieser Datenbestände. (SAP, 2014)

  • Applikationsschicht
    Auf dieser Schicht sind sämtliche Programme angesiedelt. Ganz egal, ob es sich dabei um hauseigene Anwendungen aus den entsprechenden Business Modulen handelt, oder ob es spezielle Kunden- und Partnerentwicklungen sind — Alle Anwendungen laufen auf dieser Ebene und greifen auf die Datenbankschicht zu. (SAP, 2014)

  • Präsentationsschicht
    Was hilft ein Programm, wenn nichts visualisiert werden kann? Um eine optimale Userexperience zu gewährleisten, gibt es eine Schicht auf der die Benutzeroberflächen anzutreffen sind. Hier werden letztendlich Eingaben getätigt und schließlich Ergebnisse dargestellt.

3.2. Die Anmeldung auf dem System

Abbildung 4. Anmeldescreen

In Abbildung 4 sind drei wesentliche Eingaben, die es zu verstehen gilt:

  1. Mandant (Client
    SAP-Systeme arbeiten mit einem sogenannten Mandantensystem. Dabei handelt es sich um eine dreistellige Nummer, die auch in den meisten Datenbanktabellen als Teil eines zusammengesetzten Primärschlüssels genutzt wird. Hinter dem Mandanten verbirgt sich dabei die Domäne "MANDT" mit einem "CLNT"-Datentyp mit Länge von drei Zeichen. Sinn und Zweck des Ganzen ist es, dass ein SAP-System von theoretisch mehreren Unternehmen genutzt werden kann. Jedes Unternehmen hat dabei einen eigenen Mandanten und darf in diesem Daten bearbeiten, einsehen, ändern etc. Die Daten eines anderen Mandanten können dabei nicht eingesehen oder gar modifiziert werden. Die meisten Unternehmen nutzen allerdings das Mandantenkonzept zur Unterscheidung von Werken, Filialen oder Tochterunternehmen. Ein mögliches Beispiel könnte dabei folgendermaßen aussehen:

    Mandant Werk

    100

    Werk Deutschland (Berlin)

    200

    Werk Frankreich (Paris)

    300

    Werk Spanien (Madrid)

  2. Benutzername (User)
    Hierbei handelt es sich um einen ganz normalen Anwendername mit dem ein herkömmliches Login abläuft. Dieser Name basiert dabei auf der Domäne "UNAME". Hinter dieser Dom&aouml;ne liegt ein "CHAR"-Datentyp mit einer Beschränkung auf 12 einzelne Zeichen. Groß- und Kleinschreibung wird beim Anmeldenamen nicht unterschieden. Weiterhin muss der Name mit einem Buchstaben beginnen und darf keine Umlaute oder Sonderzeichen beinhalten.

  3. Passwort
    Eine maximal 30-stellige Zeichenfolge, die sowohl Buchstaben, Ziffern und Sonderzeichen beinhalten darf. Im Gegensatz zum User wird hierbei zwischen Groß- und Kleinschreibung unterschieden.

4. Die Entwicklungsumgebung

Die wesentliche Hauptentwicklungsumgebung ist die sogenannte "ABAP Development Workbench". Die Workbench lässt sich mit der Transaktion SE80 starten. Ein wichtiges Stichwort in diesem Zusammenhang ist das "Repository". Hier landen sämtliche Entwicklungsobjekte wie zum Beispiel herkömmliche Reports, Pakete, Klassen, Strukturen etc. Diese Objekte sind dabei mandantenunabhängig, was bedeutet, dass wenn zum Beispiel auf Mandant 100 eine Klasse angelegt wird, diese auch auf Mandant 200 eingesehen werden kann. Dieses Verhalten ist anders als bei entsprechender Datenhaltung bzw. Customizing, da diese — wie bereits erwähnt — immer als Teil des zusammengesetzten Primärschlüssel den Mandanten beinhalten. Ist man also auf Mandant 100 angemeldet, so sieht man letztendlich auch nur die Daten, die im Mandanten 100 abgelegt sind. (SAP, 2014)

4.1. Suche innerhalb des Repository

Mandantunabhängigkeit impliziert eine Flut von entsprechenden Repository Objekten. Die Frage ist hier, wie man sich in dieser Masse zurechtfinden sollte. Auch hier hat die SAP vorgesorgt mit dem Repository Infosystem. Hier wird eine freie Suche nach entsprechenden Objekten ermöglicht. Es ist also möglich zum Beispiel alle Objekte eines bestimmten Entwicklers zu finden: Mehr→Umfeld→Repository Infosystem

Abbildung 10. Repository Infosystem

Weiß man allerdings nicht genau nach was man suchen sollte, so stellt die freie Suche mit Hilfe des Repository Infosystems ein Problem dar. Weiß man allerdings in welcher Anwendung zum Beispiel eine spezielle Struktur verwendet wird, so kann die SAP-Anwendungshierarchie genutzt werden. Diese Funktionalität kann dabei über die Transaktion SE81 erreicht werden. Hier werden sämtliche Anwendungen angezeigt. Sobald man eine Komponente aufklappt, erscheinen die dazugehörigen Pakete. Durch einen Doppelklick auf eines, landet man direkt wieder in der Workbench und kann sich sämtlichen Inhalt des Pakets auflisten und anzeigen lassen:

4.2. Werkzeuge der Workbench

Um schließlich ohne Probleme mit den ganzen Repository Objekte arbeiten zu können, werden weiterhin innerhalb der Workbench mehrere einzelne Werkzeuge angeboten. Die wichtigsten werden hierbei kurz erläutert:

  • ABAP Editor
    Der ABAP Editor wird hauptsächlich für die Erstellung von ABAP Quellcode genutzt. Er ist zum einen angesiedelt innerhalb der SE80, aber kann außerdem über SE38 aufgerufen werden.

  • ABAP Dictionary
    Hier werden die eigentlichen Datenbanktabellen, Views, Domänen, Strukturen etc. bearbeitet. Transaktion hierzu ist die SE11.

  • Screen Painter
    Der Screen Painter ist das wesentliche Werkzeug für die Erstellung und Verwaltung von Dynpros (Transaktion SE51).

  • Menu Painter
    Wie der Namen schon vermuten lässt, werden über dieses Tool Änderungen an Menüleiste, Symbolleiste, sowie entsprechende Funktionstastenbelegungen vorgenommen. Die Transaktion hierzu ist die SE41.

  • Function Builder
    Mit Hilfe dieses Tools können Funktionsbausteine angezeigt und entsprechend auch editiert werden. Dieses Tool kann auch über die SE37 erreicht werden.

  • Class Builder
    Natürlich darf auch ein Werkzeug für die Bearbeitung und Anlage von Klassen nicht fehlen. Transaktion ist die SE24.

Die vorherige Auflistung zeigt sehr schön, dass jedes einzelne Werkzeug als einzelnes Modul innerhalb eines SAP-Systems aufgerufen werden kann. Wesentlich komfortabler ist allerdings die Möglichkeit, welche die ABAP Workbench hierbei bietet. Alle genannten Werkzeuge sind über die Workbench erreichbar und auch integriert. Über einen Doppelklick auf jeweilige Repository Objekte kann man sofort mit der Bearbeitung loslegen!

4.3. Aufbau der Workbench

Die Workbench selbst gliedert sich in zwei wesentliche Bestandteile:

  • Navigationsbereich
    Der Navigationsbereich dient hauptsächlich zum Anzeigen von Repository Objekten. Über ihn können abhängig von der gewünschten Objektkategorie alle entsprechenden Objekte hierarchisch aufgelistet werden. Wählt man zum Beispiel "Paket" aus und gibt einen Paketnamen an, so werden alle dazugehörigen Objekte angezeigt. Über einen Doppelklick landet man dabei in der Anzeige bzw. in der Bearbeitung. Weiterhin gibt es pro Auflistungspunkt auch ein Kontextmenü, welches sich über einen Mausklick mit der rechten Maustaste öffnet. Sehr praktisch ist weiterhin die Navigation zwischen einzelnen Objekten über die jeweiligen Navigationspfeile. Um angesprochene Funktionalitäten nochmal zu verdeutlichen, dient folgende Abbildung:

    image::./Bilder/Navigationsbereich.png

  • Werkzeugbereich
    Sobald man das gewünschte Repository Objekt gefunden hat und sich dieses anzeigen lassen oder gar bearbeiten will, führt man — wie angesprochen — innerhalb des Navigationsbereichs einen Doppelklick aus. Das entsprechende Objekt landet im Anschluss innerhalb des Werkzeugbereichs, wo es abgeändert werden kann.

4.4. Pakete, Namensräme und Transportschiene

Bevor die eigentliche Entwicklung innerhalb eines SAP-Systems losgehen kann, ist es von größter Bedeutung sich vor Augen zu führen, was eigentlich ein Paket ist, wofür es Namensräume gibt und was es letztendlich mit Transporten auf sich hat. Beginnen wir hierzu zunächst mit "Paketen" bzw. sogenannten "Entwicklungsklassen".

Das Repository eines SAP-Systems ist schier unendlich groß. Sobald ein Unternehmen damit beginnt, eigene Entwicklungen auf einem System anzustellen, kommt man schnell an den Punkt, dass es ohne eine ordentliche Strukturierung unübersichtlich wird. Der Grund liegt dabei an der Fülle von Programmen, Klassen, Strukturen etc. Damit all diese Objekte geordnet werden können, gibt es eine Struktur mit Hilfe der sogenannten "Entwicklungsklassen" bzw. der "Pakete". Ein Paket ist mehr oder weniger nichts anderes als eine Art Verzeichnis in dem sich alle Repository Objekte verstecken. Die Verzeichnisse selber werden dabei nach einzelnen Modulen oder Programmen gruppiert, sodass eine maximal mögliche Übersichtlichkeit gewährleistet wird. Eine mögliche Gruppierung wäre hier zum Beispiel ein Paket für die ganze Datenbeschaffung, ein Paket für die Anwendungslogik und eines für die Benutzeroberfläche. (Färber & Kirchner, 2008)

Abbildung 12. Pakete

Sobald man ein Entwicklungsobjekt wie zum Beispiel ein Paket anlegen will, muss natürlich eine entsprechende Benennung her. Hierzu arbeitet die SAP mit Namensräumen, die registriert werden können. Grund dafür liegt in der Tatsache, dass jedes Repository Objekt im Wesentlichen global im System benutzt werden kann. Ohne jeweilige Namensräume treten hierbei natürlich Probleme auf. Entwicklungsobjekte müssen immer eindeutige Namen haben. Sobald man allerdings zwei Objekte hat, die in unterschiedlichen Anwendungen genutzt werden sollen, mehr oder weniger aber das gleiche machen, kommt es ohne Namensräume zu Namenskonflikten. Da sich auf einem SAP-System nicht immer nur Entwicklungen für einen einzigen Kunden befinden, sondern auch z. B. Standardlösungen, kann es durchaus vorkommen, dass gleiche Datentypen existieren, die auch gleich benannt werden. Aus diesem Grund bietet die SAP die Möglichkeit spezielle Präfixe zu reservieren, die im Anschluss weltweit nur von einem einzigen Kunden benutzt werden dürfen. Diese Präfixe beginnen dabei immer mit einem "/" und dürfen maximal zehnstellig sein. Für das Produkt "Bank Analyzer" wurde zum Beispiel der Namensraum "/BA1" reserviert. Es kann natürlich auch sein, dass kein Namensraum benötigt wird. Aus diesem Grund stellt die SAP den sogenannten "Z-Namensraum" bzw. "Y-Namensraum" zur Verfügung. Einen Überblick über die im jeweiligen System reservierten Namensräume gibt dabei die Transaktion SE09. Hierbei handelt es sich um den sogenannten "Transport Organizer" mit welchem Transportaufträge verwaltet werden können. Innerhalb dieses Programms gibt es die "Transport Organizer Tools", über diese können im Anschluss die registrierten Namensräume eingesehen werden. Folgende Screenshots zeigen den genauen Vorgang:

Wie angesprochen haben wir uns gerade eben im sogenannten "Transport Organizer" (SE09) aufgehalten. Diese Transaktion ist äußerst wichtig für die — innerhalb von SAP-Systemen — auffindbaren Transportschienen. Grundlegend gesehen haben Unternehmen meistens immer drei SAP-Systeme im Einsatz:

  • Entwicklungssystem
    Auf diesem System werden sämtliche Entwicklungen durchgeführt. Ganz egal ob es sich dabei um das Anlegen von Datenbanktabellen handelt oder ob hier Klassen und Methoden angelegt werden. Auf diesen Systemen haben Entwickler sogenannte Entwicklerschlüssel und dürfen ihre täglichen Codierungsarbeiten durchführen.

  • Testsystem
    Auf Entwicklungssystemen selber werden die jeweiligen Programme etc. erstellt. Auf diesen Systemen gibt es meistens jedoch nur Datenmaterial mit eher minderwertiger Qualität. Aus diesem Grund existiert in den meisten Unternehmen eine komplette Spiegelung eines Systems, wie es auch im tatsächlichen Betrieb anzufinden ist. Nachdem jeweilige Entwicklungen abgeschlossen sind, werden sämtliche Programme, Klassen etc. auf dieses System "transportiert" und dort ausführlich getestet. Im Endeffekt passiert hier nichts weiter, als dass die Fülle von Repository Objekten von einem auf das andere System verlagert wird.

  • Produktivsystem
    Das Produktivsystem ist das System, welches im täglichen Betrieb eines Unternehmens beansprucht wird. Auf ihm werden alle jeweiligen Geschäftsvorfälle und alle relevanten Unternehmensprozesse abgebildet. Sobald eine Entwicklung ordentlich getestet und festgestellt wurde, dass es keine Probleme für den Live-Betrieb gibt, wird der Stand des Testsystems auf das Produktivsystem transportiert.

Die Frage ist jetzt nur, wie denn all diese jeweiligen Repository Objekte von A nach B und von B nach C gelangen. Entscheidend hierfür sind sogenannte "Transportaufträge". Ein solcher Auftrag ist eigentlich eine Änderung für ein anderes System. An einer solchen Änderung sind in den meisten Fällen mehrere Mitarbeiter beteiligt. Damit eine maximale Übersichtlichkeit hier vorhanden ist, wird für jeden beteiligten Mitarbeiter eine eigene Aufgabe innerhalb des Auftrags erstellt. Somit kann jeder nachvollziehen, welche Repository Objekte von einem jeweiligen Mitarbeiter angelegt wurden. Dieses System hat dabei eine Reihe von Vorteilen:

  • Klare Trennung der Entwicklungsobjekte unterschiedlicher Mitarbeiter.

  • Entwicklungsobjekte können von Mitarbeitern im Auftrag bearbeitet werden. Erst nach Freigabe des Auftrags sind sie für andere Entwickler zugänglich.

  • Transportwege legen fest, welche Aufträge in welche Systeme transportiert werden. Dies geschieht alles automatisch nach anstoßen des Transports. (SAP, 2014)

Wichtig für das Verständnis ist weiterhin, dass es zwei Arten von Änderungsaufträgen gibt:

  • Workbench-Auftrag
    In einen Workbench-Auftrag landen sämtliche Repository Objekte wie Strukturen, Datenelemente, Reports, Klassen etc.

  • Customizing-Auftrag
    Ein Customizing-Auftrag umfasst wie der Name schon sagt Customizing. Also spezielle Einstellungen, die von Mandant zu Mandant unterschiedlich sind.

4.5. Pakete, Transporte — Der Weg zum ersten Report / Programm

Nachdem nun geklärt wurde, wie letztendlich die Entwicklungsumgebung aussieht, was das Repository ist und was es mit Paketen und Transporten auf sich hat, ist es Zeit erste tatsächliche Gehschritte im System zu wagen. Da alle Entwicklungsobjekte generell gegliedert und strukturiert werden sollten, wird zu Beginn ein Paket angelegt. Innerhalb der SE80 wählt man dabei unter "Repository Browser→Objekttyp Paket" aus. Im Eingabefeld darunter wird ein Name vergeben. Für unser Training hat jeder Benutzer einen eindeutigen User bekommen. Da kein spezieller Namensraum zugewiesen ist, bewegen wir uns im Z-Namensraum. Der Paketname sieht also für jeden User folgendermaßen aus: "ZXXX" wobei das XXX für die eindeutige Nummer steht, die jeder User hat. Sobald der Name eingetragen wurde und man Enter drückt, müssen entsprechende Paketattribute ausgewählt werden. Die relevanten sind im Folgenden erklärt:

Abbildung 16. Anlage eines Pakets

  1. Paket
    Hier landet der eigentliche Name des Pakets.

  2. Kurzbeschreibung
    Eine kleine prägnante Beschreibung, für was das Paket genutzt wird.

  3. Anwendungskomponente
    Wie bereits in Teilkapitel 4.1 angesprochen, beziehen sich sämtliche Repository Objekte immer auf eine spezielle Anwendung bzw. ein Modul. Hier kann zum Beispiel angegeben werden, dass das Paket zum Beispiel zum Modul SD gehört. In unserem Fall müssen wir hier allerdings nichts eintragen.

  4. Softwarekomponente
    Eine Softwarekomponente ist eine Menge von Objekten, die gemeinsam auslieferbar sind. Für Kundenentwicklungen trägt man hier immer die Komponente "HOME" ein.

  5. Transportschicht
    Bei der Transportschicht handelt es sich um eine Linie von Systemen an die transportiert werden kann. Diese Schichten bzw. Schienen geben dabei die entsprechenden Folgesysteme an.     Sollte eine Strukturierung erforderlich sein, so kann man mit Oberpaketen arbeiten. In dieses Feld wird dabei der Name des entsprechenden Pakets eingetragen.

    • Entwicklungspaket: Enthält lediglich Repository Objekte.

    • Strukturpaket: Kann nur Hauptpakete beinhalten.

4.6. Das erste Programm

Wie bereits in Kapitel 1.1 beschrieben, geht die Historie von ABAP auf das Schreiben von sogenannten Reports zurück. Hauptanwendungszweck waren betriebswirtschaftliche Auflistungen und Auswertungen mit herkömmlichen Lesezugriffen auf Daten. Anfang der 90er Jahre, mit der Einführung von R/3, ging es dann schlagartig bergauf (im funktionellen Sinne wohlgemerkt). Daten konnten nicht nur gelesen werden, sondern auch modifiziert, angelegt und sogar gelöscht werden. Heutzutage hört man immer noch von Entwicklern das Wort "Report", allerdings wird dabei meistens ein vollständiges Programm assoziiert, wie man es auch von anderen Programmiersprachen kennt. Um Reports / Programme auf einem System anzulegen, gibt es zwei wesentliche Möglichkeiten. Entweder man wählt im Navigationsbereich innerhalb der Workbench den Punkt "Programm" aus und gibt im Anschluss einen Namen ein, oder man löst es über Navigation innerhalb des Kontextmenüs. 4.7. Der Debugger

Über den Debugger kann der Report Schritt für Schritt überprüft werden. Der Debugger selbst läft dabei wieder wie alles in der SAP-Welt innerhalb eines eigenen "Debugger-Programms". Es handelt sich allerdings um ein relativ mächtiges Tool, mit dem folgende Aktionen möglich sind:

  • Debuggin im Einzelschritt (F5)
    Hierdurch wird ein Programm Zeile für Zeile ausgeführt. Werden innerhalb eines Programms zum Beispiel Unterprogramme aufgerufen, so bewirkt diese Interaktion, dass auch in die Unterprogramme hineingesprungen wird.

  • Ausführen (F6)
    Ähnlich wie Debugging im Einzelschritt allerdings mit dem Unterschied, dass zum Beispiel Unterprogramme, Funktionen etc. als ein Schritt angesehen werden. Der Debugger läuft also über die Anweisung hinweg.

  • Return (F7)
    Hierzu muss man das Debugging im Einzelschritt verstehen. Befinde ich mit innerhalb eines Unterprogramms oder einer Funktion, so interessiert mich vielleicht nur ein spezieller Teil. Habe ich diesen Teil gefunden, will ich eventuell wieder in das Hauptprogramm zurückkehren → alle verbleibenden Anweisungen des Unterprogramms werden ausgeführt und man befindet sich wieder beim aufrufenden Programm.

  • Weiter bis Cursor (Shift + F8)
    Man kann den Cursor an eine beliebe Stelle im Programm stellen und über den Menüpunkt "Debugger→Weiter bis Cursor" oder Shift + F8 zu der jeweiligen Cursorstelle springen.

  • Zur Anweisung springen
    Ähnlich wie weiter bis Cursor nur mit dem Unterschied, dass hier auch innerhalb des Programmcodes rückwärts navigiert werden kann!

  • Debugging beenden (F8)
    Hierdurch wird das eigentliche Programm schlichtweg bis zum Ende abgearbeitet und der Debugger beendet.

  • Breakpoints anlegen
    Breakpoints können zum einen innerhalb des Quellcodes selber angelegt werden. Hierzu einfach an die entsprechende Stelle navigieren und am linken Bereich der Zeilennummern einen Rechtsklick tätigen. Es öffnet sich das Kontextmenü, wobei einige Arten von Breakpoints zur Auswahl stehen:

    • Session Breakpoint
      Wie der Name schon sagt, ist dieser Breakpoint nur für die aktuelle angemeldete Usersession gültig. Sobald die Sitzung beendet wird, werden alle entsprechenden Breakpoints auch wieder gelöscht.

    • Externer Breakpoint
      Diese Breakpoints waren früher unter dem Namen "User Breakpoint" bekannt. Ähnlich wie auch Session Breakpoints bleiben sie für eine aktuelle Usersession gültig, allerdings bleiben sie auch für künftige Benutzersitzungen wie zum Beispiel RFC- oder HTTP-Requests gültig → Diese Breakpoints werden also hauptsächlich für externe Applikationen eingesetzt.

    • Debugger Breakpoint
      Hierbei handelt es sich um Breakpoints, die direkt innerhalb des Debuggers bei speziellen Anweisungen platziert werden sollen. Dadurch kann augenblicklich mit F8 an die gewünschte Stelle gesprungen werden.

    • Breakpoint Anweisung
      Es handelt sich hierbei um eine direkte Anweisung, die im Quellcode verankert wird und folglich dazu führt, dass der Debugger geöffnet wird. Die Anweisung lautet dabei: "BREAK-POINT".

  • Feldinhalte ansehen und abändern (Doppelklick auf Dictionary-Objekt)
    Durch einen Doppelklick auf die meisten verfügbaren Dictionary-Objekte innerhalb des Programms, wird der Inhalt neben dem Quelltext angezeigt. Hierbei hat man auch die Möglichkeit Feldinhalte komplett abzuändern!

Die Frage, die sich jetzt allerdings stellt ist, wie denn der Debugger eigentlich gestartet werden kann? Hierfür gibt es zwei Möglichkeiten:

  • Man setzt einen Session Breakpoint bzw. Externen Breakpoint oder eine Breakpoint Anweisung innerhalb des Quellcodes und führt das Programm ganz normal aus.

  • Rechtsklick auf den Report "Ausführen→Debugging" oder Menü "Programm→Ausführen→Debugging".

4.8. Eigene Transaktionen für Reports anlegen

Wie bereits erwähnt, interagiert man innerhalb eines SAP-Systems sehr häufig mit Hilfe von Transaktionen. Für eigene Reports bietet das natürlich auch Vorteile, da häufig Anwender innerhalb des Systems nicht immer die Berechtigungen haben, um Entwicklungswerkzeuge zu starten. Eine Transaktion selbst ist dabei nichts anderes als ein Link, über den zum Beispiel ein Report gestartet werden kann. Angelegt werden sie auch innerhalb der ABAP Workbench durch einen Rechtsklick auf das eigene Paket "Anlegen→Weitere (1)→Transaktion".
