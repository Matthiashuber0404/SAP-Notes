1. Grundlagen

Datenbanken spielen innerhalb von SAP-Systemen eine essenzielle Rolle.

Zugriff auf Datenbanktabellen aus einem Report erfolgt in ABAP mittels der integrierten Datenbanksprache Open-SQL.

SQL (Structured Query Language) ist eine Datenbanksprache, die von nahezu allen relationalen Datenbankmanagementsystemen (DBMS) benutzt wird.

Standard-SQL besteht aus folgenden Teilen:

  • Data Manipulation Language (DML): Datenmanipulationssprache, mit der Daten aus der Datenbank gelesen, geändert und gelöscht werden.

  • Data Definition Language (DDL): Datenbankdefinitionssprache, mit der Primär- und Fremdschlüssel, etc. angelegt, geändert und gelöscht werden.

  • Data Control Language (DCL): Datenbankkontrollsprache, verwalten von Zugriffsrechten auf DB-Tabellen und Datensätze.

Aber:

  • In den einzelnen DBMS (Oracle, SqlServer, DB2, MySql, PostgreSQL, …) gibt es Abweichungen vom Standard-SQL

  • SAP hat mit Open-SQL eine DBMS-unabhängige Untermenge von Standard-SQL in ABAP integriert. Über die (früher in R/3, jetzt in S/4 HANA) integrierte Datenbankschnittstelle wird Open-SQL in Standard-SQL umgesetzt.

OPEN-SQL enthält nur DML. DDL wird über das DDIC (Data-Dictionary) gelöst, DCL über das R/3-Berechtigungskonzept.

3. Die SELECT-Anweisung mit * (alle Felder)

3.1. Grundaufbau

Die Syntax ist prinzipiell folgende Anweisung (noch nicht vollständig, da sehr komplexer Befehl):

SELECT <Auswahlfelder>
  FROM <Datenbanktabelle>
  INTO <Ziel>
 WHERE <Bedingung>
 ORDER BY <Sortierfelder>.
Dabei gilt:

  • Auswahlfelder: müssen in der DB-Tabelle enthalten sein; wenn Sie alle Felder lesen wollen, müssen Sie den Stern verwenden

  • Ziel: Arbeitsbereich, Feld oder Feldleiste

  • Bedingung: optional, links muss ein Feld der DB-Tabelle stehen, rechts eine Variable o.ä. (z.B. Literal)

  • Sortierfelder: optional, stammen aus DB-Tabelle

3.2. Die SELECT-Schleife

Normalerweise werden durch die SELECT-Anweisung mehrere Sätze in der DB-Tabelle gefunden. Dadurch ergibt sich eine Schleife, die durch ENDSELECT abzuschließen ist. In jedem Schleifendurchgang wird jeweils ein Datensatz in den Arbeitsbereich gestellt. Der Arbeitsbereich (gs_teacher) ist in Bezug auf die DB-Tabelle zu definieren und ist eine Strukturvariable.

Beispiel:

Lesen Sie alle Lehrer und geben Sie diese als Liste aus.

DATA: gs_teacher TYPE zkoe2_d_teacher.
SELECT * FROM zkoe2_d_teacher
  INTO gs_teacher.
  WRITE:/ gs_teacher-id, gs_teacher-surname, gs_teacher-name, gs_teacher-office_id.
ENDSELECT.
Figure 2. Ergebnis Lehrer (mit SELECT-Schleife)
Beurteilung dieser Lösung:

Das vorherige Beispiel definiert dabei zunächst eine Struktur basierend auf einer Zeile der Datenbanktabelle ZKOE2_D_TEACHER. Im Anschluss werden über den * sämtliche Felder der jeweils aktuellen Zeile der Datenbanktabelle ZKOE2_D_TEACHER in die Struktur gs_teacher eingelesen. Zeile für Zeile wird dabei im Anschluss die ID des Lehrers sowie der Vor- und Nachname und die Büronummer ausgegeben. Es gibt jedoch einen sehr wichtigen Haken bei dieser Technik. Die Verbindung zur Datenbank bleibt während der gesamten Verarbeitung erhalten. Jeder neue Satz bedeutet einen neuen Lesezugriff auf die Tabelle. Performancetechnisch ist dies eher problematisch.

Tip
 Die SELECT-Schleife sollte daher nicht verwendet werden. Höchstens in Ausnahmefällen, wie bei sehr geringem Datenaufkommen kann eine Ausnahme gemacht werden.

3.3. SELECT in eine interne Tabelle (Array Fetch)

Um das angesprochene Problem bezüglich Performance aus dem Weg zu räumen, gibt es auch die Möglichkeit, sämtliche benötigten Daten in einem Rutsch in eine interne Tabelle zu laden und diese dann zur Laufzeit nach dem eigentlichen SELECT verarbeiten zu können.

Vorteile und Nachteile:

  • Die Datenbank muss nur einmal die Daten sammeln und diese übergeben.

  • Vorteil: Die Verarbeitung geht wesentlich schneller, da kein weiterer Zugriff auf die Datenbank erforderlich ist.

  • Nachteil: Sie müssen sich einen Arbeitsbereich für den Arbeitsspeicher definieren. Das ist heutzutage i.d.R. kein Problem mehr.

Beispiel:

Geben Sie nur die Lehrer aus einem bestimmten Büro (z.B. 103) aus.

DATA: gt_teacher TYPE TABLE OF zkoe2_d_teacher.
SELECT * FROM zkoe2_d_teacher INTO TABLE gt_teacher
  WHERE office_id = 103.
cl_demo_output=>display( gt_teacher ).
Erläuterung:

  • Definition einer internen Tabelle gt_teacher mit Typisierung TYPE TABLE OF
    Beachten Sie dabei die Benennung: g für global und t für Tabelle

  • Es gibt keine ENDSELECT-Anweisung, da die Datensätze auf einmal in die interne Tabelle geschrieben wurden.
    Damit der Array Fetch funktioniert, muss beim SELECT unbedingt INTO TABLE verwendet werden.

  • Zur Kontrolle der Daten habe ich hier eine Testklasse benutzt.
    Die Klasse cl_demo_output benutzt die statische Methode display, um die interne Tabelle auszugeben und dient in erster Linie zum Kontrollieren, ob die gelieferten Daten stimmen.

Figure 3. Ergbnisanzeige über Testmethode cl_demo_output⇒display()

In der Regel sollen die Daten aber als Liste ausgegeben werden. Dann müssen Sie diese in einer LOOP-Schleife verarbeiten.
Die LOOP-Schleife wird ausschließlich zur zeilenweisen Verarbeitung von internen Tabellen benutzt.

Beispiel:

Geben Sie alle Lehrer aus, die Ihr Büro im Erdgeschoss haben (Büronummer beginnt mit 1).

SELECT * FROM zkoe2_d_teacher INTO TABLE gt_teacher
  WHERE office_id BETWEEN 100 AND 199.
LOOP AT gt_teacher INTO gs_teacher.
  WRITE:/ gs_teacher-id, gs_teacher-surname, gs_teacher-name, gs_teacher-office_id.
ENDLOOP.
Note
 In der Loop-Schleife kann die Systemvariable sy-tabix benutzt werden. Diese Systemvariable fungiert als Schleifenzähler und wird bei Start der LOOP-Schleife automatisch auf 1 gesetzt.
Figure 4. Ergebnisliste Lehrer (über LOOP-Schleife)

3.4. Ergebnis von SELECT-Abfragen

Bevor das Ergebnis von SELECT-Abfragen bzw. anderen Operationen weiterverarbeitet wird, muss erst geprüft werden, ob die Abfrage erfolgreich war.
Dazu benutzt man das Systemfeld sy-subrc.

Note
 sy-subrc = 0, wenn die Aktion erfolgreich war, ansonsten ist das Systemfeld ungleich Null (hier 4).
Beispiel:

Geben Sie alle Lehrer aus, deren Name mit C beginnt.

SELECT * FROM zkoe2_d_teacher INTO TABLE gt_teacher
  WHERE surname LIKE 'C%'.
IF sy-subrc <> 0.
  MESSAGE 'Keine Lehrer gefunden!' TYPE 'S' DISPLAY LIKE 'E'.
ELSE.
  cl_demo_output=>display( gt_teacher ).
ENDIF.
Figure 5. Fehlermeldung: Systemfeld sy-subrc <> 0

3.5. Anzahl der Ergebnisse

Manchmal benötigt man auch die Anzahl der Ergebnisse. Dann kann man das Systemfeld sy-dbcnt abfragen.

Beispiel:

Speichern Sie alle Lehrer, die den Vornamen KARIN haben, egal ob groß- oder klein geschrieben, in einer internen Tabelle. Geben Sie die gefundene Anzahl in einer Message aus.

SELECT * FROM zkoe2_d_teacher
  WHERE upper( name ) = 'KARIN'
  INTO TABLE @gt_teacher.
MESSAGE |Anzahl: { sy-dbcnt }| TYPE 'I'.
Figure 6. Systemfeld sy-dbcnt abrufen
Erläuterung:

  • Obige SELECT-Anweisung arbeitet mit der neuen ABAP-Syntax:

  • Bei der WHERE-Klausel können Einschränkungen gemacht werden. In den Beispiel-Testdaten stehen alle Vornamen case-sensitiv. Werden Werte gesucht, dann achtet man i.d.R. nicht darauf, wie man diese eingibt (Groß-Kleinschreibung). Auf Datenbankebene kann dies über upper geprüft werden.

  • Bei der neuen Schreibweise muss WHERE und INTO vertauscht werden.

  • Die Arbeitspeichervariablen (Host-Variablen) müssen mit @ maskiert werden.

3.6. Sonderfall: SELECT SINGLE

Wie der Name schon vermuten lässt, kann mit der Anweisung SELECT SINGLE ein einzelner Datensatz selektiert werden. Die Selektion soll dabei möglichst immer mit dem Primärschlüssel erfolgen!

Beispiel:

Geben Sie den Lehrer mit der ID = 12 aus.

SELECT SINGLE * FROM  zkoe2_d_teacher
  INTO gs_teacher
  WHERE id = 12.
cl_demo_output=>display( gs_teacher ).
Figure 7. Ergebnis des SELECT SINGLE
Note
 Die SELECT SINGLE-Anweisung ist perfekt geeignet, um die Existenz von Datensätzen zu prüfen.

4. Die SELECT-Anweisung mit Spaltenangaben

Für die Übersichtlichkeit und Performance Ihres Programmes ist es viel besser, wenn Sie nicht mit * arbeiten, sondern nur mit einer Feldleiste.

Die Auswahl- oder Sendeleiste enthält alle gewünschten Felder der Datenbanken.

Für den Ziel- oder Empfangsbereich gibt es verschiedene Möglichkeiten.

Beispiel:

Aus der Datenbanktabelle ZKOE_D_TEACHER sollen nur der Name und der Vorname ausgegeben werden.

4.1. Eigene Struktur

  • Definition eines Strukturtyps, die genau die gleichen Felder (in der gleichen Reihenfolge !!!) enthält wie die Spaltenliste.

  • Definition einer internen Tabelle, die sich auf diesen Typ bezieht.

TYPES: BEGIN OF ts_teacher,
         surname TYPE zkoe2_d_teacher-surname,
         name    TYPE zkoe2_d_teacher-name,
       END OF ts_teacher.
DATA: gt_teacher TYPE TABLE OF ts_teacher.

4.2. Zugriff über Spaltenliste

Abweichend von der bisherigen Vorgehensweise über SELECT * sollen nur mehr ausgewählte Spalten selektiert werden.

Nach der neuen Syntax sind diese genau so wie klassischen realtionalen Datenbanken mit Komma zu trennen.

Die neue Syntax erfordert ebenso, dass das Schlüsselwort INTO nach dem WHERE steht.

Zusätzlich können die Spalten über Funktionen (z.B: LENGTH, UPPER) direkt beim Lesen verarbeitet werden.

In diesem Beispiel sollen alle Lehrer, deren Nachname kürzer als 6 Zeichen ist, ausgegeben werden.

SELECT surname, name
    FROM zkoe2_d_teacher
    WHERE LENGTH( surname ) < 6
    INTO TABLE @gt_teacher.
  cl_demo_output=>display( gt_teacher ).
Figure 8. Lehrername maximal 5 Stellen lang

4.3. CORRESPONDING FIELDS - Angabe

Alle Lehrer, deren Nachname mit R und Vorname mit K beginnt, sollen ausgegeben werden.

 SELECT * FROM zkoe2_d_teacher
    WHERE concat( surname, name ) LIKE 'R%K%'
    INTO CORRESPONDING FIELDS OF TABLE @gt_teacher.
  cl_demo_output=>display( gt_teacher ).

CORRESPONDING bedeutet, dass nur Felder mit gleichem Feldnamen übertragen werden.

4.4. Klammertechnik

Empfangsfelder werden hinter der INTO-Klausel in Klammer angegeben

Lehrer mit der Id = 10 soll ausgegeben werden.

DATA: gv_surname   TYPE zkoe2_d_teacher-surname,
      gv_office_id TYPE zkoe2_d_teacher-office_id.
  SELECT SINGLE surname, office_id
    FROM zkoe2_d_teacher
    WHERE id = 10
    INTO (@gv_surname, @gv_office_id).
  cl_demo_output=>display( | { gv_surname } { gv_office_id } | ).
Tip
 Ideal für SELECT SINGLE

5. Joins

Programmintern können Felder aus verschiedenen DB-Tabellen ausgewählt und innerhalb einer einzigen SELECT-Anweisung zu einem Verbund zusammengefasst werden. Dies realisiert man durch eine JOIN-Anweisung in der FROM-Klausel.

Da in der Select-Anweisung nun mehrere Datenbanktabellen auftreten, müssen bei der Benennung der Felder deren DB-Tabellennamen mitangegeben werden. Den DB-Tabellennamen verbindet man über die Tilde mit dem Feldnamen (Die Tilde verwendet man für Joins, um Verwechslungsmöglichkeiten zu „normalen“ Feldern zu vermeiden!!).

Ein Alias-Name kann für jede DB-Tabelle angegeben werden. Dieser Name wird dann in der gesamten SELECT-Anweisung verwendet. Der Alias-Name dient dazu, die Anweisung übersichtlicher zu machen.

5.1. Inner Join

Ein Join-Ausdruck besteht aus einem Operator INNER JOIN, auf dessen rechter Seite immer der Name einer DB-Tabelle stehen muss. Links davon kann eine DB-Tabelle stehen (wie unten) oder ein erneuter Join-Ausdruck.

Für jeden Join-Ausdruck muss eine ON-Bedingung angegeben werden. Die ON-Bedingung besteht aus einem oder mehreren Vergleichsausdrücken, die mit AND verknüpft sein müssen.

Beispiel: Klassen mit ihren Klassleitern sollen ausgegeben werden.

TYPES: BEGIN OF ts_classinfo,
         id        TYPE zkoe2_d_class-id,
         shortname TYPE zkoe2_d_class-shortname,
         surname   TYPE zkoe2_d_teacher-surname,
       END OF ts_classinfo.

DATA: gt_classinfo TYPE TABLE OF ts_classinfo.

SELECT c~id shortname surname
  FROM zkoe2_d_class AS c
  INNER JOIN zkoe2_d_teacher as t
  ON c~ctid = t~id
  INTO TABLE gt_classinfo.
cl_demo_output=>display( gt_classinfo ).
Figure 9. Klassenlehrer

5.2. Left Outer Join

Wenn die Zeilen der linken Seite selektiert werden sollen, obwohl auf der rechten Seite die ON-Bedingung nicht erfüllt ist, muss der Left Outer Join verwendet werden.

Die Regeln für die Join-Ausdrücke sind wie beim Inner Join.

Die Spalten jeder Zeile der rechten Seite, welche die ON-Bedingung nicht erfüllt, wird mit Null-Werten gefüllt und mit den Spalten der linken Seite verknüpft, d.h. es ergibt jede Zeile der linken Seite des Left Outer Join unabhängig von der ON-Bedingung mindestens eine Zeile in der Selektion.

Beispiel: Alle Lehrer sollen ausgegeben werden, und wenn sie Klassenleiter sind, auch der Klassenname.

TYPES: BEGIN OF ts_teacher,
         surname   TYPE zkoe2_d_teacher-surname,
         name      TYPE zkoe2_d_teacher-name,
         shortname TYPE zkoe2_d_class-shortname,
       END OF ts_teacher.

DATA: gt_teacher TYPE TABLE OF ts_teacher.

SELECT surname name c~shortname
  FROM zkoe2_d_teacher AS t LEFT OUTER JOIN zkoe2_d_class AS c
  ON t~id = c~ctid
  INTO TABLE gt_teacher
  ORDER BY t~id.
cl_demo_output=>display( gt_teacher ).
Figure 10. Alle Lehrer

6. Aggregatfunktionen

Über Aggregatfunktionen (MIN, MAX, usw.) können Inhalte von Spalten verdichtet (=aggregiert) werden.

Die Anweisung : select sum( posnetto ) from auftragsposition into… . summiert die Werte der Spalte posnetto über alle Zeilen der angesprochenen Tabelle hinweg und legt das Ergebnis in einer Variablen ab.

Wir erhalten also nur einen (=1) Datensatz mit einer (=1) Ergebnisspalte.

Komprimierte Informationen über den gesamten Datenbestand:

MIN() Ermittlung des kleinsten Wertes
MAX() Ermittlung des größten Wertes
AVG() Berechnung des Mittelwertes
SUM() Aufsummierung eines bestimmten Feldes
COUNT() Anzahl der betroffenen Datensätze

Beispiele:

Als Basis wird die Datenbanktabelle SFLIGHT aus dem Flugdatenmodell verwendet.

Figure 11. Flugdatenmodell

Es wird die Flugverbindung DL-0106 betrachtet (carrid = 'DL', connid = '0106').

COUNT: Anzahl aller Flüge
SELECT COUNT(*) FROM sflight INTO @DATA(ln_count2)
   WHERE carrid = 'DL' AND connid = '0106'.
WRITE:/'Insgesamte Anzahl an Flügen für Flugverbindung DL-0106:',(3) ln_count2.
MIN: frühester Flug
SELECT MIN( fldate ) FROM sflight INTO @DATA(lv_date1)
  WHERE carrid = 'DL' AND connid = '0106'..
WRITE:/'Der früheste Flug ist am', lv_date1 DD/MM/YYYY.
MAX: spätester Flug
SELECT MAX( fldate ) FROM sflight INTO @DATA(lv_date2)
  WHERE carrid = 'DL' AND connid = '0106'..
WRITE:/'Der späteste Flug ist am', lv_date2 DD/MM/YYYY.
AVG: durchschnittliche Sitzbelegung
DATA ln_avg TYPE p.
SELECT AVG( seatsocc ) FROM sflight INTO @ln_avg
  WHERE carrid = 'DL' AND connid = '0106'.
WRITE:/'Durchschnittliche Sitzbelegung für Flugverbindung DL-0106:',(4) ln_avg.
SUM: Zahlungssumme
DATA ln_sum TYPE p DECIMALS 2.
SELECT SUM( paymentsum ) FROM sflight INTO @ln_sum
  WHERE carrid = 'DL' AND connid = '0106'.
WRITE:/'Zahlsumme für Flugverbindung DL-0106:',(13) ln_sum.

7. Gruppierung

Zusätzlich können wir eine Gruppierung von Zeilen vornehmen: hierüber werden gleichlautende Inhalte der angegebenen Gruppierungsspalte zu einer gemeinsamen Gruppe zusammengefasst.

Die Anweisung select auftrnr, sum( posnetto ) from auftragsposition group by auftrnr into… führt dazu, dass die Aggregation für jede Auftragsnummer (Spalte auftrnr) getrennt durchgeführt wird.

Wir erhalten jetzt also tendenziell viele/mehr Zeilen und durch die (quasi-)verpflichtende Nennung der Gruppierungsspalte (=auftrnr) auch mehrere Spalten.

Exemplarisch resultieren folgende Werte:

Auftrnr

sum(posnetto)

12323

1400 EUR

12343

2500 EUR

23442

9334 EUR
Note
 Technisch könnten wir auf die Ausgabe der Gruppierungsspalte auftrnr verzichten, z.B. dann, wenn das Ergebnis im Rahmen einer weiteren Unterabfrage verarbeitet wird. Das ist aber eher die Ausnahme und für uns nicht relevant.

Wenn Sie weitere Spalten für die Bildung einer (gemeinsamen) Gruppe verwenden wollen, dann notieren Sie diese sowohl in der Spaltenliste wie auch bei der group-by – Klausel.

Regeln:

(1) Enthält die SELECT-Klausel neben Aggregatfunktionen noch normale Spaltenangaben, müssen diese in der GROUP BY-Klausel aufgeführt werden. Sämtliche Spalten, die nicht in der GROUP BY-Klausel aufgeführt sind, müssen mit Aggregatfunktionen zusammengefasst und ein einziger Wert berechnet werden.

Beispiel:
Alle Klassen sollen zuzüglich der Anzahl der Schüler pro Klasse ausgegeben werden.

SELECT cl~id, cl~shortname, cl~longterm, COUNT(*) AS count_pup
  FROM zkoe2_d_class AS cl INNER JOIN zkoe2_d_pupil AS pu
  ON cl~id = pu~cid
  INTO TABLE @DATA(gt_class)
  GROUP BY cl~id, cl~shortname, cl~longterm
  ORDER BY cl~id.

cl_demo_output=>display( gt_class ).
Figure 12. Alle Klassen mit den Schülerzahlen

(2) Wenn die Anzahl der mit der GROUP BY-Klausel zusammengefassten Zeilen durch Bedingungen eingeschränkt werden sollten, dann kann die HAVING-Klausel verwendet werden.

Die having-Klausel wendet Bedingungen auf gruppierte Datensätze an. Die Bedingungen müssen nicht denen der Auswahlliste entsprechen. Die Verwendung der having-Klausel setzt einen group-by-Ausdruck voraus.

Beispiel:
Nur Klassen, die weniger als 3 Schüler haben, sollen ausgegeben werden.

SELECT cl~id, cl~shortname, cl~longterm, COUNT(*) AS count_pup
  FROM zkoe2_d_class AS cl INNER JOIN zkoe2_d_pupil AS pu
  ON cl~id = pu~cid
  INTO TABLE @DATA(gt_class2)
  GROUP BY cl~id, cl~shortname, cl~longterm
  HAVING COUNT(*) < 3
  ORDER BY cl~id.

cl_demo_output=>display( gt_class2 ).
Figure 13. Klassen mit Schülerzahlen < 3
Note
 Wird die aufnehmende Tabelle inline definiert, dann muss den Aggregaten umbedingt ein Alias zugewiesen werden.

Achtung:

Wann verwenden Sie where, wann group by ?

Die Gruppierung ist DB-technisch ein außerordentlich aufwändiger Vorgang, da die Datensätze zu sortieren und zu Gruppen zusammenzufassen sind.

Wenn Sie 3 Mio. Zeilen gruppieren, Aggregate berechnen und dann über having nur auf die Bedingung „auftrnr = 234234“ abprüfen (5 Rohdatensätze), haben Sie 99,99% unnütze DB-Arbeit verlangt.

Tip
 Verwenden Sie having ausschließlich für Bedingungen, die auf Aggregate anzuwenden sind. Verwenden Sie where für Bedingungen, die schon auf die Rohdatensätze angewandt werden können.
