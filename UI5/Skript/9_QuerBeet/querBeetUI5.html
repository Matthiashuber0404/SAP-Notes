3. Daten filtern

3.1. Server oder Client?

Ein Filter kann für den Client oder den Server gesetzt werden.

Standardmäßig ist er im Backend (Server) gesetzt.

Auszug aus der manifest.json
"models": {
    "i18n": {
        "type": "sap.ui.model.resource.ResourceModel",
        "settings": {
            "bundleName": "ch04.chap04_json01.i18n.i18n"
        }
    },
    "": {
        "type": "sap.ui.model.odata.v2.ODataModel",
        "settings": {
            "defaultOperationMode": "Server",
            "defaultBindingMode": "OneWay",
            "defaultCountMode": "Request"
        },
        "dataSource": "SCHOOL",
        "preload": true
    }
},

Dies bedeutet, dass bei Verwendung des Filters immer eine Anfrage an den Server ausgelöst wird, welcher der Filter mitgegeben und in der Methode xxxSet_get_entityset verarbeitet wird.

Dies ist dann sinnvoll, wenn sich die Daten oft ändern und man immer die aktuellen Werte am Client benötigt. Der Nachteil davon ist, dass die Performance stärker belastet wird.

Soll auf dem Client gefiltert werden, dann ist der defaultOperationMode auf Client zu setzen.

Wird nun ein Filter verwendet, erfolgt keine Anfrage auf dem Server, sondern es wird nur mit den bisherigen Daten gearbeitet.

Der Vorteil liegt in der schnelleren Verarbeitung, ist aber nur sinnvoll, wenn der Datenbestand unverändert ist.

Erstellung eines Suchfeldes
<SearchField id="searchClass" width="20%" search="onSearch"></SearchField

Bei Auslösen des Filters muss der Filter aufgebaut und an das zu filternde Objekt gebunden werden. Im vorliegenden Beispiel wurde mit einer Tabelle gearbeitet.

Implementierung der Behandlermethode
onSearch: function(oEvent) {
    var sQuery = oEvent.getParameter("query");
    var aFilter = [];
    if (sQuery) {
        aFilter.push(new Filter("Stufe", FilterOperator.Contains, sQuery));
    }
    var oTable = this.byId("tPupils");
    var oBinding = oTable.getBinding("items");
    oBinding.filter(aFilter);
},

4.2. Zusammenhang mit EDM-Coretyp Date Time

Figure 3. Entity pupil
Erklärung

Das Datum wird bei Entity immer in der Form yyyy-MM-ddT00:00:00 erwartet. Bei normalen Eingabefeldern, z.B. <Input id="testdat" type="Date" /> erfolgt die Speicherung immer nur 8-stelling(yyyymmdd) und muss für die Übergabe noch extra aufbereitet werden.

7. SAP-User auslesen

In den <head> der index.html müssen folgende Skripte eingefügt werden:

    <script src="/sap/public/bc/ui2/services/sap/ui2/srvc/error.js"></script>
    <script src="/sap/public/bc/ui2/services/sap/ui2/srvc/utils.js"></script>
    <script src="/sap/public/bc/ui2/shell-api/sap/ui2/shell/startup.js"></script>

In der Regel wird der Benutzer gleich bei Start der Webseite geholt. Deshalb wird im Controller des Start-Views in der init-Methode der Benutzer ausgelesen.

onInit: function() {
    var that = this;
    var oUser = sap.ui2.shell.getUser();
    oUser.load({}, function() {
        var sapbenutzer = oUser.getId();
        console.log("SapUser " + sapbenutzer);
        that._setUser(sapbenutzer);
    }, function() {
        alert("Error")
    });

},
_setUser: function(sapbenutzer) {
    // SapBenutzer in geeignetem ViewFeld anzeigen
},
Hier handelt es sich um eine asynchrone Funktion. Sinnvollerweise sollte unter der load-Funktion keine Anweisung stehen, welche auf das Funktionsergebnis angewiesen ist.

12. Tabelle dynamisch erzeugen

Zunächst muss eine leere Tabelle im View angelegt werden.

<Table
    headerText="Neue Schüler"
    id="tPupils"
    growing="true" growingScrollToLoad="true" alternateRowColors="true">
</Table>

Vorliegender Code erzeugt für die Tabelle mit Zeilen mit je 4 Spalten.

onInit: function() {
    var tUeb = ["Nachname", "Vorname", "Klasse", "Gruppe"];
    oTable = this.getView().byId("tPupils");
    var s = 4;
    for (var i = 0; i < s; i++) {
        var oColumn = new sap.m.Column("col" + i, {
            header: new sap.m.Label({
                text: tUeb[i]
            })
        });
        oTable.addColumn(oColumn);
    }

    for (var i = 0; i < 10; i++) {
        var oCell = [];
        for (var j = 0; j < s; j++) {
            var cell1 = new Input({
                    value: "",
                    id: "f_" + tUeb[j] + "_" + i,
                    editable: true
                }
                //erstes Element hat die id="f_Nachname_0-inner"
            );
            oCell.push(cell1);
            }
            aColList = new sap.m.ColumnListItem("aColList_" + i, {
                cells: oCell
            });
            oTable.addItem(aColList);
        }
    }
}

13. Mehrere Daten übertragen

13.1. POST

Grundvoraussetzung
Daten müssen in einem Objektarray gespeichert sein.
Im vorliegenden Beispiel liegen die Daten im Array tPupils und werden von vorigen Punkt (Tabelle dynamisch erzeugen) übernommen.

Speicherprozess durchführen:

onInsert: function() {
    console.log("...speichere");
    var tPupils = [];
    for (var i = 0; i < 10; i++) {
        if ($("#f_Nachname_" + i + "-inner").val() !== "") {
            var pupil = {
                "Id": 1,
                "Surname": $("#f_Nachname_" + i + "-inner").val(),
                "Name": $("#f_Vorname_" + i + "-inner").val(),
                "Cid": parseInt($("#f_Klasse_" + i + "-inner").val()),
                "Groupno": $("#f_Gruppe_" + i + "-inner").val()
            };
            tPupils.push(pupil);
        }
    }

    var sUrl = "/sap/opu/odata/sap/ZKOE1_PUPIL_SRV/";
    var aDataModel = new sap.ui.model.odata.ODataModel(sUrl);
    var sPath = "/pupilSet/";
    for (var i = 0; i < tPupils.length; i++) {
        var newData = tPupils[i];
        var batchChanges = [];
        batchChanges.push(aDataModel.createBatchOperation(sPath, "POST", newData));
        aDataModel.addBatchChangeOperations(batchChanges);
    };
    aDataModel.submitBatch(function(data) {
            console.log(data);
        },
        function(oError) {
            console.log(oError);
        });
}
