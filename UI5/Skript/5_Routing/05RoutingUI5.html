1. Begriff

Bisher wurde nur ein View verwendet. Wenn die Anwendung größer wird, ist es sinnvoll, den Inhalt auf mehrere Seiten (Views) aufzuteilen.

Dann ist eine Navigation zu anderen Seiten nötig. In UI5 spricht man von Routing.

Es gibt verschiedene Arten von Routing.

2. Overview-Detail

2.1. Planung der Route

Sinnvoll ist es, sich zuerst eine Site-Map zu erstellen (Plan, welche Views es geben soll und wie die Navigation verläuft).

Figure 1. Site Map

Die App-View fungiert als Shell, in den der jeweilige View nach Aufruf gestellt wird, das heißt, bei Anlage eines neuen Projekts muss dies der Start-View sein.

In den App-View wird als erster View der Overview-View gesetzt. Von hier aus kann in den Detail-View geroutet werden.

2.2. Voraussetzungen

Zunächst ist der Router zu initialisieren, was bei Anlage des Projekt i.d.R. bereits automatisch erfolgt.

Component.js
// enable routing
this.getRouter().initialize();

Diese Anweisung wird benötigt, um das Routing zu erlauben.

Der App-View sollte folgenden Inhalt haben:

<mvc:View controllerName="routing.template.controller.App1" xmlns:mvc="sap.ui.core.mvc" displayBlock="true" xmlns="sap.m">
	<Shell id="shell">
		<App id="app">
			<pages>
				<Page id="page" title="Titel">
					<content></content>
				</Page>
			</pages>
		</App>
	</Shell>
</mvc:View>

Im gleichen Namensraum wird nun ein neuer View angelegt (z.B. durch Kopie eines anderen Views):

Anlegen Overview
<mvc:View  xmlns:mvc="sap.ui.core.mvc" xmlns="sap.m"
	controllerName="routing.template.controller.Overview">
	<App>
		<pages>
			<Page title="{i18n>title}">
				<content>
					<Button text="zur Detailseite" press="onDetail" />
				</content>
			</Page>
		</pages>
	</App>
</mvc:View>
Note
 Das <Shell>-Objekt fehlt hier.

2.3. Route anlegen

Momentan wird die Overview noch nicht angezeigt. Sie muss erst in die Shell platziert werden.
Dazu öffnen Sie die manifest.json.
De manifest.json beinhaltet außer den Routen auch sämtliche Einstellungen, Datenquellen und Modelle, welche für das Projekt gelten.

"sap.ui5": {
	"flexEnabled": false,
		"rootView": {
			"viewName": "routing.template.view.App1",
			"type": "XML",
			"async": true,
			"id": "App1"
		},
		...
}

Kontrollieren Sie, ob der rootView der App-View (hier App1) entspricht.

Ergänzen Sie jetzt die routes und targets mit der Overview-Page.

"routes": [
	{
		"name": "RouteApp1",
		"pattern": "RouteApp1",
		"target": [
			"TargetApp1"
		]
	},
	{
		"name": "OverviewRoute",
		"pattern": "",
		"target": [
			"Ov"
		]
	}
],
"targets": {
	"TargetApp1": {
		"viewType": "XML",
		"transition": "slide",
		"clearControlAggregation": false,
		"viewId": "App1",
		"viewName": "App1"
	},
	"Ov": {
		"viewType": "XML",
		"viewName": "Overview"
	}
}

Unter targets muss ein Eintrag für jeden View angelegt werden. Das Attribute viewName muss dem tatsächlichen Namen der View entsprechen (ohne die Endung .view.xml).

Unter routes werden die eigentlichen Navigations-Routen hinterlegt.

Bei Starten der Seite wird nun die Overview-Page angezeigt.

Figure 2. Anzeige Overview-Page in der Shell von App-View

Bei Wahl des Buttons zur Detailseite soll auf die DetailPage geroutet werden.

Dazu legen Sie nun einen zweiten View Detail an.

Das XML soll zunächst nur einen Text beinhalten.

<mvc:View xmlns:mvc="sap.ui.core.mvc" xmlns="sap.m" controllerName="routing.template.controller.Detail">
	<App>
		<pages>
			<Page title="{i18n>Detail}">
				<content>
					<Text text="Anzeige der Details"></Text>
				</content>
			</Page>
		</pages>
	</App>
</mvc:View>

Dann muss die manifest.json um eine weitere route und ein weiteres target erweitert werden:

"routes": [
    ...
    {
		"name": "DetailRoute",
		"pattern": "Detail/{detailPath}",
		"target": [
			"Det"
		]
	}
],
"targets": {
    ...
	"Det": {
		"viewType": "XML",
		"viewName": "Detail"
	}
}

Für pattern muss in geschweiften Klammern ein Navigationsparameter für die Detail-Seite eingetragen werden.

Note
 Der View, welcher als Erster angezeigt werden muss, enthält keinen pattern. Alle anderen View müssen pattern enthalten!
Figure 3. Detail-Page

2.4. Beispiel mit Schul-Model

Es sollen auf der Overview-Page alle Klassen angezeigt werden.

Bei Auswahl einer Klasse sollen auf der Detail-Page die Schüler dieser Klasse angezeigt werden.

2.4.1. Vorbereitung

1) Das JSON-Dokument school.json muss in das Verzeichnis model kopiert werden.
2) Das Javascript-File models.js benötigt folgende Erweiterung:

createSchoolModel: function () {
	var oModel = new JSONModel();
	oModel.loadData("model/school.json");
	return oModel;
}

3) In Component.js muss die Methode createSchoolModel aufgerufen und das Model "school" gesetzt werden.

this.setModel(models.createSchoolModel(), "school");

Nun können die Daten aus dem JSON-Model verwendet werden.

2.4.2. Overview-Page

View
Zur Klassenanzeige wird das XML-Tag <List> verwendet.

<List id="idClassList" items="{school>/CLASS}"
	mode="SingleSelect" selectionChange="onDetail">
	<StandardListItem
		title="{school>CDESC}"
		description="{school>CID}"
		icon="sap-icon://activate">
	</StandardListItem>
</List>
Figure 4. Overview: Alle Klassen
Erläuterung:

1) Mit path wird der Zugriff auf die Model-Daten ermöglicht (school>/CLASS).
2) Durch mode wird die Anzeige eines Knopfes pro Listeintrag sowie die Art der Auswahl festgelegt. Hier soll immer nur eine Klasse ausgewählt werden können (mode="SingleSelect").
3) Das Attribut selectionChange bewirkt das Auslösen der Methode onDetail, wenn ein Knopf gedrückt wird. Die Detaildaten der ausgewählten Zeile werden der Methode mitgegeben und können im Controller weiterverarbeitet werden.

Controller
Im Controller ist die Methode onDetail zu codieren:

onDetail: function (oEvent) {
	var obj = oEvent.getSource().getSelectedItem(); //1)
	var oRouter = new sap.ui.core.UIComponent.getRouterFor(this); //2)
	oRouter.navTo("DetailRoute",{detailPath:obj.getDescription()}); //3)
}
Erklärung:

zu 1)
Die Methode oEvent.getSource().getSelectedItem() holt das ausgewählte Objekt.

In der Console sieht dies folgendermaßen aus (Teilausschnitt):

Figure 5. Ergebnis aus oEvent.getSource().getSelectedItem()

Das Objekt besitzt ein Objekt mProperties, welches die gewünschten Daten beinhaltet.

Auf Properties kann man immer mit der Methode getProperty(parameter) zugreifen. in diesem Fall sieht der Aufruf folgendermaßen aus: getProperty("\description");

Der komplette Aufruf liefert dann die Klasse (hier: FS12).

zu 2)
Es soll geroutet werden. Deshalb muss zunächst ein Router-Objekt angelegt werden.

zu 3)
Es wird die Methode navTo aufgerufen. Dem Router wird außerdem der pattern (hier die classId) mitgegeben.

2.4.3. Detail-Page

An den aktuellen Pfad wird der Parameter gebunden. Am Ende der Adresse steht die jeweilige Klasse. Die Adressleiste sieht in etwa folgendermaßen aus:

View
Die ausgewählte Klasse soll nochmal angezeigt werden.

Anschließend werden die Schülerdaten in einer Table dargestellt.

<content>
	<Text class="ueb" text="Folgende Klasse wurde ausgewählt:"></Text>
	<VBox class="sapUiSmallMargin">
		<l:VerticalLayout class="sapUiContentPadding" id= "idClass">
			<l:HorizontalLayout>
					<Label text="Schultyp: "></Label>
					<Text text="{x>/Sid}"></Text>
				</l:HorizontalLayout>
			<l:HorizontalLayout>
					<Label text="Kurzbezeichnung: "></Label>
					<Text text="{x>/Cid}"></Text>
				</l:HorizontalLayout>
			<l:HorizontalLayout>
					<Label text="Langbezeichnung:"></Label>
					<Text text="{x>/Desc}"></Text>
			</l:HorizontalLayout>
		</l:VerticalLayout>
	</VBox>

	<VBox class="sapUiSmallMargin">
		<Text class="ueb" text="Schüler der Klasse"></Text>
		<Table id="idPupilTable" items="{school>/PUPIL}">
			<columns>
					<Column><Label text="Schüler-Id"></Label></Column>
					<Column><Label text="Nachname"></Label></Column>
					<Column><Label text="Vorname"></Label></Column>
					<Column><Label text="Ort"></Label></Column>
				</columns>
			<items>
				<ColumnListItem type="Active">
					<cells>
						<Text text="{school>PID}"/>
						<Text text="{school>SURNAME}"/>
						<Text text="{school>FIRSTNAME}"/>
						<Text text="{school>CITY}"/>
					</cells>
				</ColumnListItem>
			</items>
		</Table>
	</VBox>
</content>

Controller

onInit: function () {
	var oRouter = new sap.ui.core.UIComponent.getRouterFor(this); // 1)
	oRouter.getRoute("DetailRoute").attachPatternMatched(this._onObjectMatched, this);  // 2)
	this.getView().setModel(this.getView().getModel()); // 3)
},
_onObjectMatched: function (oEvent) {
	/* Tabelle füllen */
	var cid = oEvent.getParameter("arguments").detailPath; // 4)
    var aFilter = [];
	if(cid){
	    aFilter.push(new Filter("CID",FilterOperator.EQ, cid));
	} // 5)
	var oList = this.getView().byId("idPupilTable"); // 6)
	var oBinding = oList.getBinding("items");  // 7)
	oBinding.filter(aFilter); // 8)

	/* Einzeldaten füllen */
	var oModel=this.getView().getModel("school");
	var oClasses = JSON.parse(oModel.getJSON()).CLASS;
	var oSel = oClasses.find(e=>{ return e.CID === cid});

	var oAuswahl = {
		"Cid": cid,
		"Desc": oSel.CDESC,
		"Sid": oSel.SID
	};
	var oAuswahlModel = new JSONModel();
	oAuswahlModel.setData(oAuswahl);
	this.getView().setModel(oAuswahlModel,"x");
}
Erläuterung:

init()
zu 1)
Zunächst muss ein Router-Objekt angelegt werden. zu 2)
Dann muss der Parameter, welcher von Ovreview an Detail übergeben wurde, gelesen und an die private Methode _onObjectMatched übergeben werden. zu 3)
Dann sind die Models zu holen (getModel()) und im aktuellen View zu setzen (setModel()).

_ onObjectMatched(): Die Methode reagiert auf den gewählten Wert und setzt auf die Schülerliste einen Filter.
zu 4)
oEvent.getParameter("arguments").detailPath entspricht der ausgewählten Klasse.
zu 5)
Mit dem Parameter wird ein Filter angelegt, der auf Gleichheit zwischen dem Parameter und der Klasse (im PUPIL-Objekt) abgleicht.
zu 6)
Die Schülerliste mit der id="idPupils" wird als Objekt in der Variable oList gespeichert.
zu 7)
Es wird das Binding zur Aggregetion der Liste ("items") geholt.
zu 8)
Der Filter wird auf die Liste angewendet.

Figure 6. Detail-Page: Schüler einer bestimmten Klasse

3. Split-App

3.1. Prinzip

Eine weitere Entwurfstechnik für Websites ist die sogenannte Split-App.
Dabei wird der Bildschirm in mehrere Bereiche unterteilt: MasterView und DetailView.

Figure 7. Split-App (Quelle: https://sapyard.com/sapui5-routing-in-a-master-detail-app/, abgerufen am 7.8.20)

Die einzelnen Views können bei Bedarf noch weiter unterteilt werden.

3.2. Vorbereitungen

Kopieren Sie das Schulmodell (school.json) in den model - Ordner. Dann erzeugen Sie ein Model (ohne Namen).

3.3. Anlage Projekt mit Start-View

Ein neues Projekt wird angelegt.
Name des Views: Splitapp
Die Shell enthält nur das XML-Tag <Splitapp> mit der id="splitapp".

<Shell id="shell">
	<SplitApp id="splitapp"></SplitApp>
</Shell>

Der Controller von Splitapp bleibt leer.

3.4. Anlage Master-View

Die veschiedenen Schulen sollen im Master-View anhand einer <List> dargestellt werden.

View
Füllen Sie den content mit folgendem Inhalt:

<List headerText="Schulen"
    items="{path: '/SCHOOL'}"
    mode ="SingleSelect"
    selectionChange="onDetail" >
	<items>
		<StandardListItem title="{SNAME}"/>
	</items>
</List>

Controller

onInit: function () {
	this.onView1();
},
onDetail: function (oEvent) {
	var oList = oEvent.getSource().getSelectedContexts(true);  // (1)
	var Path = oList[0].sPath;          // (2)
	var nummer = Path.split("/");       // (3)
	var auswahl = parseInt(nummer[2]);  // (4)
	this.onView2(auswahl);
},
onView1: function () {
	var oRouter = UIComponent.getRouterFor(this);
	oRouter.navTo("route1", false);
},
onView2: function (number) {
	var oRouter = UIComponent.getRouterFor(this);
	oRouter.navTo("route2", {
				detailPath: number
    });                                 // (5)
}

Erläuterung:
onInit(): Beim Start wird die Methode onView1() aufgerufen. Hier wird der View DetailStart gesetzt.

onDetail(): Sobald eine Schule angeklickt wurde, muss das ausgewählte Objekt ausgelesen werden.

Um das entsprechende Objekt später in der DetailSchool anzeigen zu können, muss der Pfad ausgelesen werden (1).

Figure 8. Pfad ermitteln

Da zuvor Fachschule gewählt wurde, wird als Teil des Pfades 0 zurückgegeben.
Den kompletten Pfad (sPath:"/SCHOOL/0") erhält man durch (2).
Um die Zahl zu extrahieren, splittet man Path zunächst (3). Es entsteht ein Array.
Die Zahl ist dann das zweite Element, aber ein String.Dieser muss noch in eine zahl umgewandelt werden (4).

onView2(): Der übergebene Wert wird für die Übergabe beim Routing verwendet.

3.5. Anlage Detail-Views

Es sollen zwei Detail-Views angelegt werden.

Detail-Start-View

Bei Start der Seite soll die Detailseite noch keine Informationen ausgeben, lediglich einen Infotext.

Figure 9. Master-Detail-Start

View

<Page title="Start">
	<content>
		<Text text="Bitte wählen Sie die gewünschte Schule aus!"></Text>
	</content>

Detail-School-View

Nach Auswahl einer Schule sollen in der Detail-View die genauen Daten angezeigt werden.

							    </f:fields>
						    </f:FormElement>
						    <f:FormElement label="Langbezeichnung">
							    <f:fields>
								    <Text text="{LNAME}"/>

Controller

onInit: function () {
	var oRouter = new sap.ui.core.UIComponent.getRouterFor(this);
	oRouter.getRoute("route2").attachPatternMatched(this._onObjectMatched, this);   // (1)
},
_onObjectMatched: function (oEvent) {
	this.getView().bindElement({
				path: "/SCHOOL/" + oEvent.getParameter("arguments").detailPath
    });  // (2)
}

Erläuterung:
(1) Die mitgegebene Route wird ausgelesen und an die private Methode _onObjectMatched übergeben.

(2) Der Pfad wird wieder zusammengesetzt und mit Hilfe der Methode bindElements()) an den aktuellen View gebunden.

3.6. Routing in der manifest.json

Im routing ist die Zeile "controlAggregation": "pages" zu löschen, da wir jetzt mit einer Split-App arbeiten.

		"routing": {
			"config": {
				"routerClass": "sap.m.routing.Router",
				"viewType": "XML",
				"async": true,
				"viewPath": "ch05.chap05_splitapp.view",
				"controlId": "splitapp",
				"clearControlAggregation": false
			},
			"routes": [
				{
					"name": "route",
					"pattern": "",
					"target": [
						"Master" , "DetailStart"
					]
				}, {
					"name": "route1",
					"pattern": "navDetailStart",
					"target": [
						"Master" , "DetailStart"
					]
				}, {
					"name": "route2",
					"pattern": "navDetailSchool/{detailPath}",
					"target": [
						"Master" , "DetailSchool"
					]
				}

			],
			"targets": {

				"Master": {
					"viewType": "XML",
					"viewName": "Master",
					"controlAggregation" : "masterPages"
				},
				"DetailStart": {
					"viewType": "XML",
					"viewName": "DetailStart",
					"controlAggregation": "detailPages"
				},
				"DetailSchool": {
					"viewType": "XML",
					"viewName": "DetailSchool",
					"controlAggregation": "detailPages"
				}
			}
		}

Erläuterung:
Beim routing müssen immer mehrere targets angegeben werden, weil auf einer Seite zwei Views dargestellt werden.
HINWEIS: Es können auch mehr als zwei Views angezeigt werden!

Die targets besitzen jetzt, je nach ihrer Funktionalität, eine controlAggregation, die entweder masterPages oder detailPages sein kann.
